## 一、项目定位与目标

`ChatServer` 是一个基于 C++ 实现的**高并发、分布式、微服务化实时聊天服务器后端**。它面向生产级消息收发场景，支持用户连接管理、在线状态同步、点对点/群组消息路由、历史消息持久化及跨服务协同，适用于中大型即时通讯系统（如企业 IM、游戏聊天、社交 App 后端）。

核心设计原则：
- ✅ **异步非阻塞 I/O**：基于 `Boost.Asio` 构建可伸缩网络层；
- ✅ **服务解耦**：通过 `gRPC` 实现 `ChatService` 与 `StatusService` 等横向服务通信；
- ✅ **多实例容错**：依赖 `Redis` 实现分布式锁（`DistLock`）、在线状态共享、会话广播；
- ✅ **数据分层持久化**：热数据（在线状态、消息队列）走 Redis，冷数据（用户资料、消息历史）落 MySQL。

## 三、关键模块详解

| 模块 | 文件 | 核心职责 | 技术亮点/风险提示 |
|------|------|-----------|---------------------|
| **`AsioIOServicePool`** | `include/AsioIOServicePool.h`, `src/AsioIOServicePool.cpp` | 管理多个 `boost::asio::io_context` 实例，配合线程池实现负载均衡；避免单 `io_context` 成为性能瓶颈。 | ✅ 支持自定义线程数；⚠️ 需确认 `post()`/`dispatch()` 调用是否线程安全，避免跨 context 访问 session。 |
| **`CServer` & `CSession`** | `include/CServer.h`, `include/CSession.h`, `src/CServer.cpp`, `src/CSession.cpp` | `CServer`: TCP 监听器；`CSession`: 封装 socket 生命周期、读写缓冲区、心跳超时、消息解析（可能基于 protobuf）。 | ⚠️ `CSession` 析构时必须显式 `close()` socket 并 `cancel()` pending ops，否则 Asio 可能 crash。 |
| **`ChatServiceImpl`** | `include/ChatServiceImpl.h`, `src/ChatServiceImpl.cpp`, `proto/message.proto` | gRPC Service 实现类；将 RPC 请求（如 `SendRequest`）转发至 `LogicSystem` 处理，并返回 `SendResponse`。 | ✅ 使用 `ServerAsyncResponseWriter` 实现流式响应；⚠️ 同步方法（如 DB 查询）需 offload 到线程池，防止阻塞 gRPC completion queue。 |
| **`LogicSystem`** | `include/LogicSystem.h`, `src/LogicSystem.cpp` | 业务逻辑中枢；协调用户认证、消息分发策略（单聊/群聊/广播）、状态变更通知（→ Redis/→ StatusService）。 | 🔑 是扩展性关键：新增功能（如“已读回执”）应在此层注入，而非侵入 `ChatServiceImpl`。 |
| **`UserMgr` & `MsgNode`** | `include/UserMgr.h`, `src/UserMgr.cpp`, `include/MsgNode.h`, `src/MsgNode.cpp` | `UserMgr`: 内存级用户会话注册表（`std::unordered_map<uid, CSession*>`），支持快速查找在线用户；`MsgNode`: 消息内存节点（可能含 sender/receiver/timestamp/payload），用于构建链表或环形缓冲区。 | ⚠️ `UserMgr` 读写需加锁（`shared_mutex`？）；`MsgNode` 若为裸指针链表，需严格管理生命周期防 dangling pointer。 |
| **`RedisMgr` & `DistLock`** | `include/RedisMgr.h`, `src/RedisMgr.cpp`, `include/DistLock.h`, `src/DistLock.cpp` | `RedisMgr`: 封装 hiredis 或 redis-plus-plus，提供连接池、命令执行封装；`DistLock`: 基于 `SET key value NX PX timeout` 实现可重入/自动续期分布式锁。 | ✅ 应使用 Lua 脚本保证解锁原子性（`if redis.call("get",KEYS[1]) == ARGV[1] then return redis.call("del",KEYS[1]) else return 0 end`）；⚠️ 锁过期时间需远大于业务最大耗时。 |
| **`MysqlMgr` & `MysqlDao`** | `include/MysqlMgr.h`, `src/MysqlMgr.cpp`, `include/MysqlDao.h`, `src/MysqlDao.cpp` | `MysqlMgr`: 连接池管理（如 `mysqlcppconn`）；`MysqlDao`: 数据访问对象，封装 SQL（如 `INSERT INTO messages (...) VALUES (...)`）。 | ✅ 应支持事务（`START TRANSACTION` → `INSERT` → `UPDATE` → `COMMIT`）；⚠️ 防 SQL 注入（强制使用参数化查询）。 |
| **`ConfigMgr`** | `include/ConfigMgr.h`, `src/ConfigMgr.cpp`, `config.ini` | 加载 `config.ini` 中的配置项（`server.port`, `mysql.host`, `redis.url`, `grpc.status_service_addr` 等），提供全局只读访问接口。 | ✅ 支持 INI 格式解析（`#include <boost/property_tree/ini_parser.hpp>`）；⚠️ 需验证必填项是否存在，缺失时应 panic 或 fallback。 |
| **`StatusGrpcClient` & `ChatGrpcClient`** | `include/StatusGrpcClient.h`, `src/StatusGrpcClient.cpp`, ... | 封装 gRPC Channel 和 Stub，提供同步/异步调用接口（如 `GetOnlineStatus(uid)`），用于跨服务协作。 | ✅ Channel 应启用 keepalive；⚠️ 异常处理需覆盖 `DEADLINE_EXCEEDED`, `UNAVAILABLE`, `CANCELLED` 等状态并降级（如缓存旧状态）。 |

## 五、潜在优化与风险点（建议后续验证）

1. **【高优】连接泄漏风险**<br>
   - 检查 `CSession::~CSession()` 是否调用 `socket_.close()` 并 `cancel()` 所有 pending async operations。<br>
   - 验证 `AsioIOServicePool::stop_all()` 是否被正确调用（进程退出时）。

2. **【中优】分布式锁安全性**<br>
   - 审计 `DistLock.cpp`：是否使用 Lua 脚本解锁？是否支持锁续期（`refresh()`）？是否记录锁持有者 trace_id？

3. **【中优】gRPC 线程模型匹配**<br>
   - 查看 `ChatServiceImpl` 中 RPC 方法是否全部 `async`（推荐）？若含同步 DB/Redis 调用，是否 `post()` 到 `AsioIOServicePool` 线程避免阻塞？

4. **【低优】配置热更新**<br>
   - `ConfigMgr` 当前是否仅启动时加载？可扩展为监听 `config.ini` inotify/fswatch，触发 `MysqlMgr::Reconnect()` 等。

5. **【可观测性】日志与指标缺失**<br>
   - 项目暂未发现 `spdlog` / `glog` 或 Prometheus client 集成 —— 建议在 `CSession`（连接数）、`LogicSystem`（QPS/延迟）、`RedisMgr`（command latency）埋点。
